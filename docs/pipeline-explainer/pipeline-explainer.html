LIC DSF pipeline

a short visual overview
A short visual walkthrough of how the Excel reverse-engineering pipeline works, how we run or export formulas, and where we are in the work plan. One idea per slide.

The problem

Excel workbooks are powerful — but the logic is inside the spreadsheet. We want the same logic in Python so economists can run scenarios without opening Excel.
The LIC DSF template is a big Excel workbook. To use it you have to open it, change cells, and look at the results. We want that same "change inputs, get outputs" behavior as a Python library.

The solution in one sentence

We trace what each cell depends on, translate the formulas to Python, and ship a library that runs the same math.
Trace dependencies, translate formulas, ship a library. That's the whole pipeline in one line.

Three layers

Stacked layers: each has one job. We only draw the picture once; then we build it.
Excel goes into excel-grapher, which traces and translates. That goes into lic-dsf-programmatic-extraction, which configures and shapes it for the LIC DSF. Out comes py-lic-dsf, the library economists use.

Layer 1: What does this cell need?

For any cell we care about (e.g. a stress-test result), we ask: which other cells does its formula use? We repeat until we hit numbers or text. That gives us a dependency graph.
excel-grapher's main job is target-driven dependency tracing. Give it a target cell; it follows the formula references until it reaches leaves. The result is a graph of nodes (cells) and edges (depends-on).

Smallest example: the two-cell demo

flowchart LR A1["A1 = 10"] B1["B1 = A1 × 2"] A1 --> B1
B1 depends on A1. The graph is two nodes and one edge.
In the two-cell demo we have A1 equals 10 and B1 equals A1 times 2. So B1 depends on A1. The dependency graph is just two nodes and one edge.

Two ways to run it

FormulaEvaluator: keep the graph in memory and evaluate formulas on it (like a tiny Excel engine).
CodeGenerator: write Python code that does the same math, so we can ship that code and no engine.
We have two execution paths. The FormulaEvaluator interprets the graph at runtime. The CodeGenerator emits Python source. Both must give the same answers—we enforce that with tests.

Same answer

Two-cell demo Evaluator: run graph → B1 = 20
Generated code: run Python → B1 = 20
We test that both paths match. That's our parity contract.
For the two-cell workbook, both the evaluator and the generated code produce 20. We require that for every cell we care about—parity between the two paths.

Layer 2: Shape it for economists

Raw export is Excel-shaped (cell addresses, one function per cell). This repo adds labels (e.g. "Real GDP Growth"), grouped inputs into setters, and annotations from the LIC DSF Guidance Note.
lic-dsf-programmatic-extraction discovers targets, enriches the graph with human-readable labels, uses RAG to annotate logic from the guidance note, and groups inputs into setters so users don't deal with raw cell addresses.

Layer 3: The library

py-lic-dsf is what economists use: a context with setters for inputs and compute_* functions for the stress tests and risk rating. No Excel, no pipeline code—just Python.
The public library is entirely generated. We don't hand-edit it. Setters live on the context; you call compute_* to get results. Self-contained and dependency-clean.

Phase 2 in one slide

Scale to all stress tabs and the risk rating · Organize ~200 setters (e.g. namespaced subclasses) · Document which inputs are needed for which outputs · Golden-master verification · Shorter, more readable generated code (graph compression, segmentation)
Key Phase 2 deliverables: full coverage of tabs, better API organization, input-output documentation, verification framework, and conciseness strategies like graph compression and segmentation.

One thing to remember

We turn Excel's dependency graph into Python and shape it into a library. The rest is detail.
Trace → translate → configure → ship. One idea per layer.
The mental model: dependency graph to Python, then shape for the user. Everything else—evaluator vs codegen, labels, RAG, setters—is how we do that one thing well.
