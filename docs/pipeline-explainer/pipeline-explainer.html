LIC DSF pipeline

a short visual overview
A short visual walkthrough of how the Excel reverse-engineering pipeline works and how we mechanically trace and translate formulas to Python.

The problem

Excel workbooks are powerful — but the logic runs inside the application. We want the same logic in Python so economists can run scenarios without opening Excel.
The LIC DSF template is a big Excel workbook. To use it you have to open it, change cells, and look at the results. We want that same behavior as a Python library.

The problem

Manual Excel work is laborious, time-consuming, and error-prone. Workflows cannot be typed, tested, or even audited. It can't be automated or done at scale.
Manual work in Excel doesn't scale and is hard to verify. We need a programmatic path.

The solution in one sentence

We trace what each cell depends on, translate the formulas to Python, and ship a library that runs the same math.
Trace dependencies, translate formulas, ship a library. That's the whole pipeline in one line.

But that's a big challenge

The LIC DSF:

  • has over 800,000 cells, including 200,000 formula cells
  • calls 55 unique functions, nested up to 7 levels deep
  • uses lookups and offsets tightly coupled to Excel's data model
The LIC DSF template is huge: hundreds of thousands of cells and formulas, with deep nesting and Excel-specific constructs.

Option 1

Drive Excel from Python. Pros: modest speedup, familiar tooling, correctness guarantees. Cons: most of the limitations of Excel still apply.
Automating Excel from Python keeps correctness but doesn't escape Excel's limits.

Option 2

Have AI agents translate to Python. Pros: leverages the "bitter lesson," improves as AI models do. Cons: unpredictable output shape, no correctness guarantees, unreproducible for new template versions.
Pure AI translation is flexible but unreliable and not reproducible across template versions.

Option 3

Translate to Python mechanically. Pros: predictable output shape, correctness guarantees, reproducible for new template versions. Cons: hard to get right, output is still Excel-shaped.
Mechanical translation gives us predictability and correctness; we can post-process to make it more Python-shaped.

Making Option 3 tractable

AI agents help make Option 3 tractable. And we can post-process the output to make it more Python-shaped.
We use AI to build the mechanical pipeline; then we shape the output into a proper library.

The lesson

For interpretability, reproducibility, and correctness guarantees, don't AI-generate the code. AI-generate the code that generates the code.
The code economists use is generated by our pipeline, not by an AI directly. That gives us control and verification.

Replicate in Python

It's mostly a solved problem to replicate this BADO + engine approach in Python.

Workbook becomes a dictionary; application becomes an emulator that evaluates over that dictionary.

Workbook → Python dictionary

Every cell is an entry with its formula and value keyed to the address of the cell.
Python 
xlsm_contents = {
    "Sheet1!A1": {
        "formula": "=B1+C1",
        "value": 5
    }
}
The in-memory representation is a dict from cell address to formula and value.

The engine: AST parser

The engine is an abstract syntax tree (AST) parser: it breaks down an Excel formula into a branching tree of operations and translates each into a Python equivalent. We build on the formulas library.
Parsing and evaluating formulas is done via an AST; we reuse existing work where possible.

Simple example AST

flowchart TB A1["A1"] Add["+"] ref_B1["B1"] ref_C1["C1"] ref_B1 --> Add ref_C1 --> Add Add --> A1
B1 and C1 feed into Add; result goes to cell A1.

Hard parts

  • If we only want particular output cells, how do we limit extraction to only the relevant inputs?
  • What sequence do we compute the cells in?
We solve these by building a graph of relationships between cells.
Dependency graph gives us both scope and order.

Dependency graph

For any cell we care about (e.g. a stress-test result), we ask: which other cells does its formula use? We repeat until we hit numbers or text. That gives us a dependency graph.
Target-driven: start from outputs and follow references to leaves.

Target-driven dependency tracing

Give excel-grapher a target cell; it follows formula references until it reaches leaves. The result is a graph of nodes (cells) and edges (depends-on).
We only extract what's needed for the targets we specify.

First layer: BADO + evaluator

It already works today. We can run the workbook this way and get results.

excel-grapher has two layers: grapher + evaluator (run the graph), and exporter (emit code).

Limitations of the first layer

  • Poor separation of concerns — formulas live in the data layer, so data and economic logic are mixed.
  • Non-transparent economic logic — the model is 200,000 atomic operations; neither readable nor maintainable.
What we want: the data layer holds constants and inputs; computational logic lives as code.
Layer 1 runs but doesn't give us a clean library shape.

Second layer: the exporter

flowchart LR excel_grapher["excel-grapher"] subgraph layer1["Layer 1: BADO + engine"] grapher["grapher"] evaluator["evaluator"] end subgraph layer2["Layer 2"] exporter end excel_grapher --- layer1 excel_grapher --- layer2
The exporter turns the graph into static code and data.

What the exporter does

The exporter turns formula cells into Python functions and outputs them, along with relevant parts of the engine, as a standalone Python library. Non-formula cells stay as a Python dictionary. That gives us a clear split: data in the dictionary, logic in code.
Data vs logic separation is achieved by the exporter.

The exporter is configurable

  • Specify targets (which outputs you want) → it generates functions that produce those outputs.
  • Specify how to group inputs → it generates setters for those groups.
  • Mark cells as constants (used in the computation but not user-settable) so they stay out of the public API.
Configuration drives the shape of the generated library.

Configuration = domain knowledge

Which cells are inputs, which are outputs, which are internal constants — that's where domain knowledge belongs. For the LIC DSF template, that configuration lives in the lic-dsf-programmatic-extraction repository.
This repo encodes what we know about the LIC DSF workbook: inputs, outputs, and constants.

Where we're headed

  • Today we still have one function per formula cell.
  • Goal: one function per economic model concept.
  • We'll use static analysis and graph analysis to find groups of functions that should be collapsed. Most of that post-processing will live in lic-dsf-programmatic-extraction; some may be reusable in excel-grapher.
Next step is collapsing the 200k operations into human-meaningful concepts.

One thing to remember

We trace dependencies, translate formulas, configure and shape the output, and ship a library. The rest is detail.
Don't AI-generate the code. AI-generate the code that generates the code.
Mechanical pipeline plus configuration gives us a reproducible, verifiable library.
