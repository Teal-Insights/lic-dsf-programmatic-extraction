Python Classes

a primer for R programmers
Hey — so I wanted to put together a short walkthrough to help bridge some of the conceptual gap between R and Python. Specifically, I want to talk about classes and objects, because that's where a lot of the "why does Python do it that way?" confusion comes from. If you understand this one idea, a lot of Python will suddenly click.

The R approach

R # load a library with string functions
library(stringr)

# call a function, pass in your data
result <- str_to_upper("hello")
# → "HELLO"
The function lives in a library. You hand it your data. It hands back a result.
So in R, when you want to manipulate a string — say, make it uppercase — you import a library like stringr, and you call a function from that library. You pass your string in as an argument, and you get a new value back. The function lives in the library, separate from your data. Your data is just… data. It doesn't do anything on its own. The function acts on it from the outside.

The Python approach

Python # no import needed
greeting = "hello"

# call a method that lives ON the string
greeting.upper()
# → "HELLO"
No library import. The method lives on the string itself.
Now here's the same thing in Python. We create a string variable, greeting, and we just call dot-upper on it. No import. No library. The uppercase method lives directly on the string. That dot is doing something important — it's saying "hey, string, YOU know how to make yourself uppercase." The capability is baked into the data itself.

Same result — different philosophy

R
str_to_upper("hello")
function acts on data
Python
"hello".upper()
data has the function
Let's see them side by side. In R, the function acts on the data from outside. In Python, the data already has the function built in. Same result — totally different mental model. In R, functions and data are separate things. In Python, they're bundled together. And that bundling? That's what a class is.

What makes this possible?

"hello"
→
an instance of class str
Every string in Python is an object —
created from a class blueprint.
So where does dot-upper come from? The answer is: every string you create in Python is actually an object — a specific instance of a class called str. Think of a class as a blueprint. The str class says: "every string that gets created will have these methods — upper, lower, replace, split, and so on." When you write "hello" in quotes, Python builds an object from that blueprint, and all those methods come along for free.

The class blueprint

class str
state (data)
"hello"
methods (behavior)
.upper()
.lower()
.replace()
.split()
.strip()
…
A class bundles state (your data) with methods (what it can do).
Here's what that blueprint looks like conceptually. A class has two parts: state, which is the actual data — in this case, the characters "hello" — and methods, which are functions that know how to operate on that state. Upper, lower, replace, split — these all live inside the class definition. So when you call greeting dot upper, you're saying: "use the upper method that's part of your own blueprint to transform your own state." That's the core idea of object-oriented programming. Data and behavior, bundled together.
In R, functions do things to data.
In Python, objects know how
to do things to themselves.
That's what a class is — a blueprint that gives objects
both their data and their capabilities.
So here's the one-sentence version. In R, functions do things to data. In Python, objects know how to do things to themselves. And a class is just the blueprint that makes that possible — it defines what data an object holds, and what it can do with that data. Once you see Python through this lens, a lot of things start making sense — why you see dots everywhere, why you don't need to import as many libraries, why everything feels like it has methods attached to it. It's because in Python, almost everything is an object, built from a class.

That's the mental model.

When you see object.method() in Python,
you're seeing a class at work.
And that's really it. That's the mental model shift. Every time you see that dot notation in Python — something-dot-something — you're looking at an object using a method that its class gave it. Now let's see how we use this same idea in py-lic-dsf.

Applying it to py-lic-dsf

We store all DSA inputs in a single context object.
That object can know how to operate on itself
and how to instruct users in its own usage.
Same idea: data + behavior in one place. The context isn’t just a bag of values — it’s an object that understands its own shape.
In py-lic-dsf we take that same idea and apply it to the DSA workflow. All inputs live in a context object. Because it's a class instance, we can design it so it knows how to operate on itself and how to guide users. It's not just a passive container — it has methods that make the library easier to use and harder to misuse.

What the context can do

context
methods (examples)
.list() — list the six input buckets
.next() — which bucket to work on next (tracks completed)
.load_from_excel(path) — load all inputs from a filled template
The context knows the workflow (Input 1 → … → Output 7) and can direct the user. It also knows how to read the Excel template — that logic lives in the object, not scattered in scripts.
So we might give the context a list method that returns the six input buckets — configuration, macro_debt_baseline, baseline_financing_terms, stress_tests_standard, stress_tests_tailored_and_custom, stress_financing_and_sdr. A next method can track which buckets are already filled and tell the user which one to work on next. And we can give it a method to load all inputs from a filled-out Excel template; the context already knows the sheet layout and cell mapping, so it just knows how to do that. All of that is possible because we have an object that understands its own shape.

Each bucket is a class (or subclass)

Concept # Each bucket can have domain-specific behavior

macro_debt_baseline.set_historical_from_weo(...) # IMF API
baseline_financing_terms.required_inputs(config) # depends on debt def, etc.
Subclasses can encapsulate how to get or set their data. One bucket might know how to pull WEO historical data via the IMF API; another might know which of its inputs are required given the current workbook configuration.
Each of the six buckets is itself a class — or a subclass of a common input-bucket base. So macro_debt_baseline might have a method that sets historical data from the World Economic Outlook using the IMF API. And we can give every bucket a method that looks at the current workbook configuration — debt definition, residency vs currency, etc. — and returns which inputs in that bucket are required. So the object doesn't just hold values; it knows the rules that apply to its slice of the DSA.

Objects that validate themselves

Because each bucket knows its shape and domain rules,
it can validate the inputs we pass to it.
e.g. residual financing must not be negative; interest rates nominal vs real by debt type. Validation lives with the data it validates.
And finally, because these objects understand their own shape and the domain constraints — things we already document in the instruction sheet, like avoid negative residual financing, or nominal vs real rate conventions — we can have the objects validate the inputs we pass to them. So validation isn't a separate layer of functions; it's built into the buckets. Same idea again: the object knows how to operate on itself and how to keep itself sane.

Why classes help py-lic-dsf

One context object: stores inputs, lists buckets, tracks progress,
loads from Excel. Six bucket subclasses: each knows how to
get/set its data, which inputs are required for the current config,
and how to validate. All of that is possible because we have
objects that understand their own shape.
So to tie it together: we have one context object that holds all the inputs and knows how to list the buckets, tell you what's next, and load from a filled template. We have six bucket subclasses that each know how to populate themselves — including from external APIs where it makes sense — which of their inputs are required given the configuration, and how to validate. All of that comes from the same idea we started with: in Python, objects know how to do things to themselves. Here we're just designing those objects so that their "shape" and "rules" are first-class, which solves our library design problems in one place.
